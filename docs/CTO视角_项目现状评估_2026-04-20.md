# CTO 视角项目现状评估：破誓圣骑士

文档日期：2026-04-20

## 1. 当前项目目标是什么？

当前项目目标更接近：

**可玩的作品集 Demo**

不是长期运营项目，也不只是一个纯技术测试原型。

更准确地说，它现在处于：

- 有明确主题表达的作品集 Demo
- 已经超过“单纯验证手感”的原型阶段
- 但还没有达到“完整可交付小型成品”的程度

### 当前定位判断

从现有 PRD 和实现状态看，这个项目的核心目标不是“先把通用 Roguelike 做完整”，而是：

- 做出一个围绕“守誓 / 破誓 / 腐化失控”展开的差异化成长 Demo
- 让玩家和面试官能快速理解这个项目的创新点
- 用较短流程完成一次有主题、有成长、有展示性的战斗体验

### 结论

当前最合理的定位是：

**一个以游戏策划面试展示为目标的、可玩的主题化肉鸽 Demo。**

当前项目已经有：

**可玩的产品外壳 + 初步主题成长系统**

但还没有形成：

**完整一局、完整构筑、完整创新机制闭环**

## 3. 目前是否已经形成完整一局游戏闭环？

### 当前能跑通到哪一步

目前已经能跑通以下流程：

1. 进入主菜单
2. 点击开始游戏
3. 进入战斗场景
4. 玩家移动、自动攻击、刷怪、生存
5. 击杀敌人获取经验
6. 升级后出现三选一
7. 选择守誓 / 破誓 / 通用成长
8. 本层计时结束后生成出口门
9. 玩家靠近出口门并按 `E`
10. 进入下一层
11. 自动保存
12. 重复上述流程

### 当前闭环缺失在哪

目前缺少“完整一局”的后半段：

- 没有明确的总层数结束点
- 没有最终结算页
- 没有失败结算页
- 没有完整的局内 build 结算回顾
- 没有明确告诉玩家“这一局已经完成”

### 实际判断

所以当前项目已经具备：

**单层循环闭环**

和：

**多层推进闭环**

但还不具备：

**完整 Run 闭环**

### 结论

目前项目属于：

**能持续往下打的爬塔 Demo**

而不是：

**真正完成一整局并结算的完整游戏流程**

## 4. 当前版本最好玩的地方是什么？最不好玩的地方是什么？

### 4.1 当前最好玩的地方

当前版本最好玩的，不是单纯“攻击手感”，而是：

**升级选择已经有身份感和主题感。**

具体体现在：

- 升级时不是普通数值三选一，而是明显分成守誓 / 破誓 / 通用
- 誓约值和腐化值让成长方向有了明确表达
- 楼层推进、出口门、阶段标题，让体验不像测试场，而更像一个有结构的 Demo
- 一部分分支技能已经能把“守誓更稳、破誓更狠”做出初步区分

这部分已经能支撑你对外讲：

“这个项目的重点不是普通割草，而是成长选择的意义改变了。”

### 4.2 当前最不好玩的地方

当前最不好玩的，是：

**战斗本体还不够丰富。**

问题主要有：

- 玩家基础战斗行为单一
- 敌人行为模型单一
- 中期以后主要是在重复刷时间
- 构筑差异还不足以显著改变局内打法
- 腐化高了以后，战斗并没有明显进入“更强但更危险”的失控状态

### 结论

当前版本的优点是：

**主题表达先立住了。**

当前版本的短板是：

**战斗内容和战斗变化还撑不起这个主题表达。**

---

## 5. 当前最缺的是什么？

如果必须在下面几项里选一个最缺的：

- 玩法深度
- 内容量
- 数值平衡
- UI
- 视觉表现
- 技术稳定性

我的判断是：

### 第一缺：玩法深度

原因是当前项目最大的风险不是“不好看”，而是：

**创新点还没有完全变成玩家在战斗中持续感知的玩法。**

目前守誓 / 破誓是成立的，但腐化失控这件事还没有真正驱动战斗节奏变化。

### 第二缺：内容量

具体表现为：

- 技能数偏少
- 敌种偏少
- 局内组合差异不够
- 中后期内容展开不够

### 第三缺：数值平衡

因为一旦玩法深度和内容量不足，数值再怎么调，也只能改善体验，不能解决“变化不够”的问题。

### 不是当前最缺的

- UI 不是当前最缺
  - 因为 UI 基本已经达到“可展示”的水平
- 视觉表现不是当前最缺
  - 因为现在更大的问题不是不够炫，而是炫的东西还没有形成明确机制后果
- 技术稳定性也不是当前最缺
  - 目前项目作为 Demo 骨架已经能跑

### 结论

当前最缺的是：

**玩法深度**

其次是：

**内容量**

## 6. 现在技能、敌人、升级系统分别做到什么程度？

### 6.1 技能系统

当前技能系统已经从“纯数值强化”迈到“组件化分支技能”阶段。

已有能力：

- 升级项数据化
- 按阵营分类出牌
- 选项可以挂接到具体技能组件
- 已有 6 个分支技能
  - 守誓 3 个
  - 破誓 3 个

当前阶段判断：

- 已经能证明“架构支持做技能分支”
- 但还远没达到 PRD 里要求的技能池规模
- 也还没形成足够强的技能联动和后期爆发感

### 6.2 敌人系统

当前敌人系统属于：

**单敌种 + 波次数值变化**

已有内容：

- 刷怪
- 追击
- 接触伤害
- 精英概率
- 血量、速度、体型倍率变化
- 关卡阶段变化

但当前明显问题是：

- 行为模型单一
- 没有远程、冲锋、召唤、区域压制等差异敌种
- 压力主要来自数量，不来自敌我互动变化

### 6.3 升级系统

当前升级系统是项目最成熟的系统之一。

已做到：

- 经验升级
- 升级弹窗暂停战斗
- 守誓 / 破誓 / 通用三类选项
- 基于誓约值 / 腐化值的权重调整
- 选择结果会修改对应阵营数值
- 选择结果可绑定到技能解锁和数值成长

当前不足：

- 第三个按钮本质还是 Common，不是更强的分流机制
- 升级后的“战斗玩法变化”还不够剧烈
- 后期升级池展开不足

### 结论

- 技能系统：架构成型，内容不足
- 敌人系统：骨架可用，玩法层次不足
- 升级系统：表达已成型，是当前最强模块之一

## 7. UI 现状如何？

### 7.1 当前 UI 整体评价

当前 UI 已经达到：

**可展示 Demo 水平**

它的优点是：

- 不再像默认 Unity 测试界面
- 已经形成一套偏统一的暗色圣域 / 腐化风格
- 主菜单、升级、HUD、阶段标题都已经有主题感

### 7.2 当前信息清晰的部分

- 当前层数清晰
- 当前血量清晰
- 誓约值 / 腐化值清晰
- 当前阶段和倒计时清晰
- 出口门提示清晰
- 升级选项的阵营归属清晰

### 7.3 当前信息不清晰的部分

- 距离下一次升级还差多少经验，不够清楚
- 当前已经拿了哪些技能，不够清楚
- 腐化升高后到底会带来什么战斗变化，不够清楚
- 当前这一层的目标是否只是“等时间到”，反馈不够明确
- 玩家死亡后发生了什么，不清楚

### 7.4 还没做的界面

- 死亡结算界面
- 通关结算界面
- Run Summary
- 技能清单 / 当前构筑面板
- 暂停 / 设置界面
- 完整的继续游戏状态提示

### 结论

当前 UI 最大问题不是“丑”，而是：

**关键信息层次还不完整。**

也就是：

“看起来已经像游戏了，但还不够像一个完整可结算的游戏。”

## 8. 代码架构是否支持快速加新技能 / 新敌人 / 新升级项？目前最大技术债是什么？

### 8.1 是否支持快速加新技能

支持，且相对顺手。

当前技能接入链路已经比较清晰：

1. 定义升级项数据
2. 在升级库里增加选项
3. 在 `LevelUpUI` 中映射效果类型
4. 在 `PlayerBranchProgression` 中挂接对应组件
5. 新组件内部处理成长逻辑

这说明：

**项目已经有“往上加分支技能”的基础架构。**

### 8.2 是否支持快速加新升级项

支持中等偏好。

因为升级项本身已经数据化，但仍有一部分逻辑写死在 `LevelUpUI` 的效果分发里。

这意味着：

- 加少量升级项没问题
- 如果升级类型快速增多，`LevelUpUI` 会越来越重

### 8.3 是否支持快速加新敌人

支持程度一般。

当前 `EnemySpawner` 更像是：

**一个针对单敌种的波次数值调度器**

而不是：

**一个真正的敌群编排系统**

所以如果你只是加“同模板换参数”的敌人，还行。

但如果你要加：

- 远程敌
- 冲锋敌
- 特殊精英
- 召唤型敌人

那当前刷怪结构就会很快开始吃力。

### 8.4 当前最大技术债

当前最大的技术债不是代码风格，而是：

**继续游戏不能完整恢复一局 build。**

也就是说，存档目前只保存了基础数值和楼层状态，没有完整保存：

- 已解锁技能
- 技能等级
- 当前构筑状态

这会直接带来一个产品层面的可信度问题：

**“继续游戏”并不是真正的继续，而只是继续数值。**

### 次级技术债

- `LevelUpUI` 职责偏重
- 玩家基础战斗和腐化失控没有真正解耦成清晰系统
- 敌人系统还没有抽象成多敌种编排结构

### 结论

架构整体上：

- 加新技能：可以
- 加新升级项：可以，但后续会变重
- 加新敌人：勉强可扩，但不够优雅

最大技术债：

**存档与构筑状态没有打通**

## 9. 当前最大的三个问题是什么？

按优先级排序如下。

### 1. 核心创新没有完全落地到战斗

这是当前最重要的问题。

项目现在已经能讲清：

- 守誓
- 破誓
- 腐化

但还没有真正让玩家在战斗中持续体验到：

**“腐化越高，越强，但也越危险、越失控。”**

如果这一点不成立，那么项目的最大卖点就会停留在文案和 UI 层。

### 2. 战斗内容太薄，导致中期体验重复

具体表现在：

- 敌人单一
- 玩家基础攻击单一
- 技能数量少
- 局内打法变化有限

结果就是：

- 前期新鲜感还可以
- 中后期内容撑不住主题野心

### 3. 完整 Run 闭环缺失

虽然现在已经能下楼、能继续，但还没有：

- 真正的失败结算
- 真正的通关结算
- 真正的完整 build 恢复

这会让项目始终停留在：

**“很完整的原型”**

而不是：

**“完整的可展示 Demo”**

### 结论

当前最大的三个问题依次是：

1. 创新机制没有完全打进战斗本体
2. 战斗内容和构筑深度不足
3. Run 闭环和结算闭环不完整

## 10. 接下来两周最合理的开发重点是什么？

### 总原则

接下来两周不要再平均发力，而要明确：

**先把“创新点能被玩出来”做实，再补内容，再补结算。**

### 建议顺序

#### 第一优先级：补完整 Run 闭环

先做这些：

- 明确总层数上限
- 完成通关结算页
- 完成死亡结算页
- 跑通返回菜单 / 继续游戏 / 重新开始
- 让继续游戏真正恢复当前 build

为什么先做：

- 这是把原型升级成完整 Demo 的关键一步
- 也是后续做演示录像和面试展示的基础

#### 第二优先级：把腐化失控做成真正战斗机制

建议至少做一个明确可展示的核心机制：

- 腐化越高，攻击越容易触发黑暗脉冲
- 或腐化越高，战斗中随机出现强化但不可完全控的暗系爆发
- 并且配清晰视觉反馈

为什么第二做这个：

- 这是项目真正的 USP
- 也是决定面试官记不记得住这个项目的核心

#### 第三优先级：扩内容深度

建议两周内至少补到：

- 再加 4-6 个升级项
- 再加 2-3 类敌人
- 让 4-7 层出现明显打法变化

为什么第三做：

- 没有内容支撑，创新点也只能短暂成立
- 中期内容必须开始撑起“构筑展开”

#### 第四优先级：做第一轮数值与信息优化

重点优化：

- 第一次升级出现时间
- 守誓 / 破誓收益感知
- 腐化增长速度
- 技能获得后的战斗体感反馈
- HUD 补 build 信息与经验信息

### 不建议这两周优先做的事

- 不建议先扩多职业
- 不建议先做复杂剧情
- 不建议先做很重的局外系统
- 不建议把大量时间花在纯装饰型 UI 上

### 两周开发重点一句话总结

最合理的顺序是：

1. 先补完整一局闭环
2. 再把“腐化失控”真正做成战斗机制
3. 再扩技能和敌人内容
4. 最后做第一轮数值和信息优化

## 最终判断

当前项目已经从“测试场原型”进化到了：

**有产品壳、有主题表达、有基础成长循环的可玩 Demo。**

但它还没有真正进化到：

**一份足够完整、足够有说服力、能稳定支撑面试展示的代表作 Demo。**

接下来最关键的，不是继续润外观，而是补齐两件事：

- 让完整一局真正成立
- 让“破誓带来力量，也带来失控”真正发生在战斗里

只要这两件事补上，这个项目就会从“有想法的 Demo”跃迁成“能被记住的 Demo”。
一、完整一局定义

1. 标准一局时长我建议定在 `10 分钟左右`，上限不超过 `12 分钟`。
原因不是内容量，而是展示节奏。
- 8 分钟偏短，容易刚进入 build 展开就结束
- 12 分钟以上对面试展示偏拖
- 10 分钟最适合做“前期理解机制 -> 中期第一次破誓爽点 -> 后期高腐化压迫 -> 结算复盘”

2. 总层数我建议定 `6 层`，不是 10 层。
原因是按展示节奏倒推，不是按 PRD 理想量倒推。
- 现在内容量撑不起 10 层
- 6 层更容易做出“1-2 层认知建立，3-4 层 build 成型，5-6 层高压展示”
- 对应每层 90 秒到 110 秒，总时长刚好落在 10 分钟上下

3. 当前每层目标我建议这版先固定为：
`活到计时结束 -> 门出现 -> 进门`
不要这版就扩清怪比例、事件完成。
原因：
- 先把 run 结构打实
- 目标越统一，玩家越容易理解这个 demo 的主轴是成长与腐化
- 差异目标适合后续内容扩充，不适合当前阶段分散资源

4. 通关最终判定我建议做：
`打完固定层数后进入通关结算`
这版先不强上 Boss。
原因：
- 现在敌人与技能深度不足，Boss 很容易变成低质量占位内容
- 固定层数结算更稳，更适合展示型 demo
- 可以把第 6 层做成“高压最终层”，用刷怪密度、精英压力、腐化视觉拉满代替 Boss

5. 死亡结算页面最低限度必须展示这些：
- 到达层数
- 存活时长
- 玩家等级
- 击杀数
- 誓约值
- 腐化值
- 阵营倾向
- 已获得技能列表
- 每个技能等级
- 最终生命状态或死因标签

我认为你列的字段里必须保留的几乎都对，唯一可以不强求的是“当前升级池状态”。

6. 通关结算和死亡结算我建议：
`同一套结构，不同情绪包装`
- 结构统一：方便开发，也方便玩家对比 run
- 情绪区分：
  - 通关偏“总结”
  - 死亡偏“审判”
这样既省成本，又能保主题表达

7. “真正恢复一局 build”最低要恢复这些字段：
- 当前层数
- 当前层剩余时间
- 当前波次阶段
- 当前等级
- 当前经验值
- 下一级所需经验
- 当前生命值
- 最大生命值
- 当前移速
- 当前攻击间隔
- 当前攻击间隔倍率
- 当前誓约值

- 已获得技能 ID 列表

- 当前击杀数

这版我不建议恢复：
- 当前场上敌人状态
- 当前升级池随机历史
- 当前地面经验球状态

二、腐化失控机制

8. 我最优先想落地的方案是：
`高腐化时，普攻概率触发黑暗脉冲`
这是第一优先，不选别的。
原因：
- 最容易接入现有自动攻击循环
- 最容易被玩家立刻感知
- 最容易做出“腐化越高越明显”的成长曲线
- 最适合做视觉展示
后续如果扩展，再叠加代价层

9. 低中高腐化的玩家体感，我希望是：
- 低腐化：稳定、安全、可控，像还在守规矩的圣骑士
- 中腐化：开始明显贪输出，爽感抬升，但你会意识到自己在往危险方向走
- 高腐化：爆发非常强，屏幕反馈明显增强，但场面开始失控，决策压力和压迫感都上来

10. 腐化代价我只选一个主代价的话，我选：
`敌人压力代价`
不是掉血，不是操作惩罚。
意思是：
- 腐化越高，脉冲越强
- 同时更容易引出更危险敌人、更高精英率、更快刷怪压力
为什么选这个：
- 最符合肉鸽体验
- 玩家不会觉得“系统故意恶心我操作”
- 风险感更像“你主动换来更危险局势”
- 更适合和爬塔结构结合

11. 腐化增长节奏我建议：
`以主动选择后的明显跳升为主，不做纯线性慢爬`
也就是“慢慢堕落”的表象要有，但真正跨阶段要靠关键抉择。
原因：
- 玩家必须对自己的堕落负责
- 这样每次破誓都更有戏剧性
- 也更利于面试时讲“这是主动风险交换，不是系统自动灌数值”

12. 腐化到阈值后的可见反馈，我希望分三层：
- HUD变化：
  - 腐化条进入新阶段时明显变色
  - 阵营倾向文案强化
  - 屏幕边缘出现轻微污染框或脉冲提示
- 玩家身上变化：
  - 角色周身出现暗色环、残影或脉冲粒子
  - 攻击瞬间附带暗焰/血色/黑环特效
- 战斗结果变化：
  - 普攻触发额外黑暗脉冲
  - 高腐化阶段精英率/敌压明显提升
  - 玩家能直接感到“输出变猛了，但战场也更危险了”

13. 我最担心的问题是：
`玩家觉得这只是白赚伤害`
这是最大风险。
其次是：
- 玩家看不懂失控从哪来
- 代价太轻，破誓变成无脑最优解
- 代价太重，玩家第一次破誓就后悔

三、现有内容拆解

14. 现有 6 个分支技能及其战斗行为变化：
- `OathDamageShield`
  - 受伤后短时间减伤
  - 改变的是容错，不改变主动打法
- `OathExpHeal`
  - 吃经验球回血
  - 改变的是拾取价值和续航路线，轻微改变走位
- `OathPurificationPulse`
  - 周期性范围脉冲伤害
  - 改变的是近身清杂能力和站位收益，算是半主动改变打法
- `FallenKillLeech`
  - 击杀概率回血
  - 改变的是滚雪球能力，偏数值收益
- `FallenLowHealthFrenzy`
  - 低血时攻速更快
  - 改变的是残血收益，鼓励冒险压血线
- `FallenDeathBurst`
  - 敌人死亡时爆炸
  - 改变的是清群节奏和连锁击杀体验，是最明确的战场行为改变之一

15. 我认为“数值强化伪装成技能”的有：
- `OathDamageShield`
- `FallenKillLeech`
- `FallenLowHealthFrenzy`

真正更能立住主题的有：
- `OathPurificationPulse`
- `FallenDeathBurst`
- `OathExpHeal` 算弱主题成立，因为它至少让“拾取=救赎”有一点行为意义

16. 如果两周内只能再补 4 个技能，我最想补这 4 个：
1. `亵渎脉冲`
- 普攻命中或攻击时概率触发黑暗脉冲
- 解决问题：把腐化核心机制真正接进战斗主循环
2. `深渊锁链`
- 攻击后弹射到附近目标
- 解决问题：提高中期清群爽感，拉开 build 差异
3. `纯净领域`
- 周期性清除部分腐化压力或压制失控触发
- 解决问题：让守誓路线不是纯防守，而是真正对抗堕落
4. `圣印惩戒`
- 低频自动圣光落点打击
- 解决问题：给守誓路线一个“稳定进攻”表达，不然守誓全是保命会太被动

17. 最该先补的 3 类敌人：
- 远程压迫怪
- 冲锋切位怪
- 高血精英锚点怪

18. 这些新敌人的作用分别是：
- 远程压迫怪：
  - 逼走位
  - 迫使玩家不能一直站桩
  - 提升后排处理优先级
- 冲锋切位怪：
  - 打断稳定节奏
  - 让高腐化高输出流付出站位代价
- 高血精英锚点怪：
  - 制造优先击杀目标
  - 提供战场中心压力
  - 让 `DeathBurst`、弹射、脉冲类技能更有发挥对象

四、确认展示型 Demo 边界

19. 我希望面试官 30 秒内记住的一句话是：
`这不是普通肉鸽加职业换皮，而是把升级变成了守誓与破誓的身份抉择，腐化越深，力量越强，战斗也越失控。`

20. 如果只能保一个卖点，我保：
`腐化失控战斗机制`
不是另外两个。
原因：
- 守誓/破誓选择如果没有战斗后果，就容易停留在文案层
- 爬塔过层结构是通用骨架，不是记忆点
- 只有腐化失控真正打进战斗，整个项目才有不可替代性

21. 这版 Demo 看起来应该做，但其实可以不做的：
- 多职业
- Boss 战
- 复杂剧情
- NPC 事件
- 长线局外成长
- 很重的装备系统
- 地图探索分支
- 太多场景主题切换
- 复杂商店系统

22. 这版 Demo 不做就不能见人的必须项，我只列 5 个：
- 完整 run 结束条件与结算
- 腐化失控核心机制
- 至少 1 条守誓路线和 1 条破誓路线能明显玩出差异
- 至少 2 到 3 类敌人形成战斗层次
- 继续游戏能真正恢复当前 build

五、7 天制作排期

23. 如果只有 7 天，我会这样排：
1. Day1：定义完整 run 结构
- 定总层数、每层时长、通关判定、死亡结算字段、通关结算字段
2. Day2：实现 run 结束与结算框架
- 死亡页、通关页、重开、回菜单
3. Day3：补完整存档字段
- 技能列表、技能等级、层内状态、击杀数
4. Day4：接腐化失控核心机制
- 普攻触发黑暗脉冲
- 阶段阈值效果
5. Day5：补 2 个关键技能
- 一个守誓，一个破誓
- 先把 build 差异拉出来
6. Day6：补 2 类关键敌人
- 远程压迫怪
- 精英锚点怪或冲锋怪
7. Day7：第一轮数值和信息调优
- 升级节奏
- 腐化节奏
- HUD 信息
- 演示流程自测

24. 这 7 天里：
- 最容易做了但没明显效果的：
  - 继续润 UI 外观
  - 因为它会让项目更像样，但不会解决玩法核心问题
- 最难做但必须做的：
  - 腐化失控机制
  - 因为它既要可感知、可解释、可平衡，还不能变成白赚伤害或纯惩罚

25. 我的明确判断：
`B. 先补腐化失控机制`
这是第一优先级。

为什么不是 A：
- 没有腐化失控，补完 run 结算后，你得到的只是一个“能结算的普通爬塔原型”
- 结构更完整了，但卖点还是没立住

为什么不是 C：
- 敌人与技能内容当然重要，但如果核心机制没立住，后面加的内容只是往一个模糊方向继续堆料
- 应该先确定“破誓到底怎么改变战斗”，再决定补什么敌人和技能去服务它

所以我的 CTO 结论很明确：

先做 `腐化失控机制`，因为它决定这个项目到底是不是“破誓圣骑士”；
然后补 `run 结算完整度`，因为它决定这个项目能不能像一个完整 demo；
最后再扩 `敌人与技能内容`，因为那是把已经成立的主线体验做厚。
